# Hackathon Winning Strategies

## Pre-Hackathon (Before Event)

### 1. Team Building
- Mix skills: frontend, backend, design, presenter
- Ensure someone is "scrappy" (can figure things out)
- Build team chemistry beforehand

### 2. Technology Prep
- Know your stack inside out
- Have boilerplate ready
- Practice integrations before

### 3. Idea Brainstorming
- Come with 5-10 ideas
- Check if it's been done
- Validate with mentors

## During Hackathon (Hour by Hour)

### Hours 0-2: Ideation & Planning
- ✅ Pick the best idea
- ✅ Define MVP (ruthlessly)
- ✅ Create architecture diagram
- ✅ Assign roles clearly

**Pro Tip**: Spend max 1.5 hours here, not 3!

### Hours 2-4: Setup
- ✅ Git repo initialized
- ✅ Environment configured
- ✅ API keys ready
- ✅ Boilerplate deployed

### Hours 4-12: Core Development
- ✅ API endpoints working
- ✅ Database seeded
- ✅ Components rendering
- ✅ First integration done

**Pro Tip**: Get something working early!

### Hours 12-20: Feature Polish
- ✅ Connect all pieces
- ✅ Handle errors gracefully
- ✅ Add UI polish
- ✅ Test flows

### Hours 20-24: Demo & Presentation
- ✅ Write demo script
- ✅ Practice 5+ times
- ✅ Fix critical bugs only
- ✅ Create README

## Winning Tactics

### Tactic 1: "Fake It Till You Make It"
- Use mock data if APIs are down
- Show potential, not just completed features
- Judges care about concept + execution

### Tactic 2: "One Feature Done > Ten Features Half-Done"
- 1 fully working feature > 5 buggy features
- Polish beats breadth
- End-to-end demo is critical

### Tactic 3: "Story > Technology"
- Don't explain the tech stack
- Explain the problem and solution
- Show impact and scale

### Tactic 4: "Differentiation Matters"
- Don't build another todo app
- Add a unique twist
- Show creativity

### Tactic 5: "Demo > Slides"
- Live, working demo beats 100 slides
- People remember what they see
- Practice so it doesn't fail
